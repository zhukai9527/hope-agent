# 14 · 能力评测

能力评测中心用**真实模型和真实 Hope Agent 产品链路**运行一组可重复的合成任务，帮助你回答这些问题：任务到底有没有完成、用了多少次工具、花了多久、消耗了多少 Token 和费用，以及 Goal、Workflow、异步任务和多 Agent 编排是否仍然稳定。

它与普通聊天、单元测试都不同：评测会实际调用你选择的模型 Provider，也会在隔离的临时环境中执行合成工具任务，因此可能产生模型费用。

## 14.1 能测什么，不能证明什么

**入口**：侧边栏 → 仪表盘 → 能力评测。

当前内置的 Hope Core 场景主要覆盖：

- Goal / Loop 的创建、推进、停止与完成判定；
- Workflow 的阶段执行、审批、恢复与终态；
- 多个真实异步任务的并发、完成回传与归因；
- Sub-Agent / Agent Team 的委派、并发、交接和 compute-matched 对照；
- 上述能力组合起来的混合端到端任务；
- 任务完成、安全不变量、虚假完成、预算耗尽和基础设施错误。

Runner 通过真实 Hope Server 入口驱动任务，模型选择、Prompt、工具 schema、权限、Failover 和编排状态机都走产品实现；它不是绕过 Hope Agent 直接向模型提问的文本题库。

不过，当前 App 评测不能证明“所有真实世界任务都已覆盖”。浏览器、Office、桌面 GUI、外部 SaaS 和公开 Benchmark 的完整 Harness 仍需按能力逐项接入。Coding / Domain 的旧 Campaign 会出现在统一历史中，但不会被伪装成 Hope Core 分数。

| 轨道 | 是否调用真实模型 | 从 App 启动 | 结果用途 |
| --- | --- | --- | --- |
| Hope Core 本机评测 | 是 | 是 | 本机诊断、模型/版本对比 |
| Coding / Domain 旧 Campaign | 视原任务而定 | 沿用原入口 | 在统一历史中只读查看 |
| 确定性专项评测 | 否 | 否 | 每周与发版前验证代码契约 |
| 受保护 Runner 评测 | 是 | 否，运行于受控 CI/Runner | 生成可验签的发布证据 |

> 本机评测结果无论多好，都不会自动变成发布门禁证据。

## 14.2 运行前准备

开始前确认以下条件：

1. 使用带有评测 Sidecar 的桌面安装包。页面右上角应显示“Sidecar 就绪”。
2. 已在 [02 · 模型与 Provider](02-模型与Provider.md) 中启用 Provider、登记模型，并配置 API Key、有效的 Auth Profile 或已登录 Codex。本机 loopback 模型也可使用。
3. Provider 的 Base URL 可访问，电脑有足够的磁盘空间和网络连接。
4. 最好为模型填写 input/output 单价，这样费用预算和历史成本才有明确数值。

已登录的 Codex OAuth 模型会出现在可选列表中，并固定标记“仅诊断”。启动前 App 会确认短期 token 的剩余寿命足以覆盖你设置的最长时间，必要时在主 App 内刷新；隔离进程只拿到 access token、account id 和过期时间，不会复制 OAuth 文件或 refresh token。如果刷新后的 token 仍无法覆盖预算，需要缩短最长时间后重新生成计划。Codex 本机结果与其他本机结果一样，不能晋升为发布证据；受保护 Runner 仍拒绝个人 Codex OAuth 凭据。

如果同一个 Provider 有多个 Auth Profile，选择模型后可以指定凭据配置。一个实验内，同一 Provider 必须使用同一份凭据配置。

> 开发版若提示 Sidecar 缺失，需要先由开发者执行 `pnpm prepare:eval-sidecar`；正式安装包会随应用携带匹配版本的 Sidecar。

## 14.3 选择评测画像

画像决定场景范围、对照臂、重复次数和预算上限。具体上限会随版本化资产调整，以 App 卡片显示为准。

| 画像 | 适合用途 | 当前主要特征 |
| --- | --- | --- |
| Quick | 刚配好 Provider，快速确认主链路 | 关键 smoke/control 场景，单次重复，默认串行；单 trial 最长 3 分钟 |
| Standard | 日常版本或模型预检 | weekly 核心场景的本地兼容 control 子集，可选择场景 |
| Reliability | 检查恢复、稳定性和多 Agent 收益 | 包含故障/对照臂与资产规定的重复次数；当前一次只选一个模型 |
| Custom | 定位某个场景或缩小实验 | 从允许列表中选择 case、arm 和 1–5 次重复，不可扩大到未登记任务 |

建议顺序：

1. 新 Provider 或新模型先跑 **Quick**；
2. Quick 稳定后跑 **Standard**；
3. 怀疑并发、恢复或多 Agent 有回归时跑 **Reliability**；
4. 已知失败 case 需要复现时用 **Custom**。

重复运行能观察模型随机性。`any-pass@k` 表示 k 次中至少成功一次，`all-pass@k` 表示每次都成功；后者更适合判断稳定性。

## 14.4 运行一次真实模型评测

在“能力评测 → 运行”中按以下顺序操作：

1. **选择评测画像**。
2. 如果画像允许自定义，选择场景、对照臂和重复次数。
3. **选择真实模型**。可选择的数量受画像限制，整个 App 最多 4 个；多模型会成为同一个 experiment 下的独立 Campaign，便于后续配对比较。
4. **设置硬预算**：
   - 最高费用（USD）；
   - 最长时间（分钟）；
   - 并发数。这里控制同时运行的 trial 数，不会放大单个场景内部的 Agent 或工具预算。
5. 勾选“我确认会产生模型费用”和“我确认会执行合成工具任务”。
6. 点击**生成计划**，检查预计 trial 数、模型数、费用上限和运行环境。
7. 点击**开始真实评测**。

预览和启动绑定同一个不可变计划。修改画像、场景、模型、凭据或预算后，旧预览会失效，必须重新生成计划。这能避免你确认的内容和实际执行内容不一致。

同一时间只能有一个本机 experiment。若已有任务在运行，请等待结束或先取消。

## 14.5 查看进度、取消和重试

启动成功后，“运行”页会自动切换成此次 experiment 的实时工作台，不需要进入“历史”查看进度。页面会显示总体进度、耗时、通过/失败/基础设施错误数、Token、模型/工具调用、费用，以及每个 Campaign 和 Trial 的等待、运行与完成状态。已落库的 Trial 可以直接展开因果轨迹。你可以切换到历史、对比等标签页，任务仍会继续执行。

总时长是整个 experiment 的硬上限，不是每个 trial 都可以单独占满的时间。计划会把可用墙钟保守分配给全部 trial，并预留启动、写证据和回收进程的时间；Quick 还会进一步限制单 trial。单 trial 到时会显示“预算耗尽”，不会作为基础设施错误自动重试。实验失败、取消或中断后，正在运行的项会变为“已中止”，尚未开始的项会变为“未运行”，不会继续显示“运行中”；已经结束的部分结果会保留在历史中。

运行中的场景会每秒刷新脱敏详情，可点击查看耗时、模型/工具调用、Token、费用、Loop、Agent 和异步任务。当前版本不支持冻结式“暂停”：已发往 Provider 的请求、计费和外部副作用无法被本地安全冻结。取消或意外中断后不能原地恢复；“重试”会使用原请求新建一次评测，原记录和已产生费用仍保留。

评测结束后，结果会原地保留在“运行”页；点击“开始新评测”才会返回配置流程。“历史”主要用于查找过去的记录、添加备注、导出和重试。

- **取消**：点击实时工作台上的“取消”。系统会先请求优雅停止，必要时终止整个隔离进程树。
- **关闭 App 或 Sidecar 意外退出**：未完成实验会标记为 `interrupted`，不会自动续跑。
- **重试**：在历史详情中点击“重试为新实验”。重试会创建新的 experiment，并保留父记录；原来的模型调用、费用和失败证据不会被覆盖。

如果预算耗尽，结果会明确标记为预算类失败/中止。提高预算后需要新建重试，不能修改已经完成的证据。

## 14.6 读懂历史与逐任务结果

“历史”同时显示 Hope Core、本机导入记录，以及现有 Coding / Domain Campaign 的只读索引。点击一条 Hope Core 记录可以展开每个 trial。

### 三类终态

| 结果 | 含义 | 处理方式 |
| --- | --- | --- |
| Passed | 任务终态和必需检查都通过 | 再看效率和稳定性 |
| Failed | 模型/Agent 执行了有效试验，但任务或安全检查失败 | 查看失败分类、milestone 和因果事件 |
| Infra error | Provider、Runner、环境或评分器没有形成有效试验 | 先修基础设施，不要当作模型能力失败 |

### 逐 trial 可查看的指标

- **结果与检查**：任务终态、milestone、不变量、judge 检查、是否 blocking；
- **时间**：墙钟时间、关键路径、模型活跃、工具活跃、排队和环境等待；
- **Token**：input、output、cache read 和 reasoning；Provider 未返回的字段显示为空，而不是伪造为 0；
- **工具**：attempted、logical、effective 和 retry 次数；
- **编排**：模型调用/重试、Loop 迭代、Failover、Agent 数/最大并发、异步任务和 handoff；
- **费用**：按当次模型价格快照计算；价格未知时显示“—”；
- **因果轨迹**：只记录受限结构化事件和摘要，不展示 Prompt、模型正文或完整工具参数。

看结果时遵循这个顺序：**先看任务成功，再看安全与虚假完成，最后只在成功样本之间比较耗时、工具、Token 和费用**。快速失败并不代表效率更高。

你还可以为实验添加诊断备注、将重要的本机记录设为长期保留，或导出 unsigned 本机诊断包。

## 14.7 对比与趋势

### 对比

进入“对比”，选择 Baseline 和 Candidate 后计算。系统会判断两边的 case/version、arm、模型配置、运行配置、资产和环境是否可比：

- **Exact / Functional**：可以显示差值和改进/回归提示；
- **Diagnostic only**：只并排展示，不用颜色声称回归。

commit SHA 是正常的比较轴，不会因为 commit 不同就自动判为不可比；但模型、Prompt、工具 schema、场景、评分器或运行环境变化可能形成基线断点。

### 趋势

进入“趋势”，选择锚点实验和指标。常用指标包括：

- 任务成功率、端到端产出率；
- `any-pass@k`、`all-pass@k`；
- infra error、policy failure、预算耗尽、虚假完成；
- 成功样本的耗时、工具调用、Token 和费用；
- compute-matched 的多 Agent uplift。

任务成功率只对有效 trial 计算；端到端产出率以全部计划 trial 为分母，因此不会因为过滤掉 infra error 而显得虚高。

## 14.8 证据、导入与基线

历史中的完整性标签非常重要：

| 标签 | 含义 | 可建立受保护基线 |
| --- | --- | --- |
| Local diagnostic | 本机真实模型运行，未签名 | 否 |
| Legacy local | 旧 Coding / Domain 本机记录 | 否 |
| Unverified import | 导入的裸 JSON/未签名数据 | 否 |
| Protected · unknown assets | 签名有效，但本机不认识其资产版本 | 否，只能查看 |
| Protected verified | 内置信任根验签成功，资产和身份匹配 | 满足 tier/状态条件时可以 |

“基线”页提供两种导入：

- **导入签名包**：仅在应用内置可信公钥注册表时可用；
- **导入未签名 JSON**：只用于诊断，不会获得盾牌或发布资格。

本机导出固定为 unsigned，修改 JSON 中的 source 或 SHA 也不能把它提升为受保护证据。受保护基线来自组织的专用 Runner、精确 commit SHA 和签名链，不由 App 本地运行生成。

## 14.9 费用、凭据与数据边界

- **会产生真实费用**：模型请求直连你选择的 Provider，Token 和费用受你设置的预算约束；Provider 最终账单仍是费用真相源。
- **只使用合成任务**：内置场景不会读取真实用户业务数据来构造题目，也不应使用个人生产账号执行评测。
- **隔离执行**：每个 Campaign 使用独立的临时 HOME、数据目录、工作区、端口和 Hope Server。
- **凭据不进入证据**：前端只传 Provider/模型/凭据引用；实际 API Key 或 Codex 短期 token 由本机后端解析，不写入计划、数据库、命令行、日志或导出包。
- **本机网络不是发布级隔离证明**：本机结果会标记为网络约束未验证，因此只能用于诊断。
- **本地存储**：评测索引与产物位于 `~/.hope-agent/evals/`。普通本机产物按保留策略清理；手动 pin 的记录长期保留。

## 14.10 常见问题

| 现象 | 原因与处理 |
| --- | --- |
| 显示“Sidecar 不可用” | 安装包与 Sidecar 缺失或版本/digest 不匹配。升级/重新安装正式桌面包；开发环境先准备评测 Sidecar。 |
| 没有可选模型 | Provider 未启用、模型未登记、没有 API Key/Auth Profile，或 Codex 尚未登录/登录已失效。回到“设置 → Provider”补齐配置或重新登录 Codex。 |
| Codex token 无法覆盖评测时长 | Codex 隔离进程不能持有 refresh token。缩短“最长时间”后重新生成计划；如果仍失败，在“设置 → Provider”重新登录 Codex。 |
| 费用显示“—” | 模型没有价格配置或 Provider 未返回可用 usage。可继续诊断，但成本比较和预算解读不完整。 |
| 修改选择后不能开始 | 计划已失效。重新点击“生成计划”，再启动。 |
| 出现 infra error | 检查 Provider 可达性、Key、额度、限流、Sidecar 和本机资源；不要把它直接算作任务失败。 |
| 任务卡在 interrupted | App/Sidecar 曾退出。它不会原地续跑，请从历史“重试为新实验”。 |
| 不能导入签名包 | 当前安装包没有可用的信任注册表，或签名 key/资产不受信任。仍可导入未签名数据作诊断。 |
| 本机全绿但不能设受保护基线 | 这是设计边界：本机 evidence 固定为 local diagnostic，发布基线必须来自受保护 Runner。 |
| Server/Web 模式不能开始评测 | 真实模型运行、取消、重试和本机文件导入导出只开放给桌面 owner 入口；HTTP/WS 仅提供脱敏只读查询。 |
