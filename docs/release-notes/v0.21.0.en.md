# Hope Agent 0.21.0 — Built-in Manual, Capability Evals, and Durable Streaming

> [简体中文](https://github.com/shiwenwen/hope-agent/blob/v0.21.0/docs/release-notes/v0.21.0.md) · **English**

> Released: 2026-07-20

> See [`CHANGELOG.md`](https://github.com/shiwenwen/hope-agent/blob/v0.21.0/CHANGELOG.md#0210---2026-07-20) for the full list.

v0.21.0 is a consolidation release built around two themes: knowing how to use Hope Agent, and being able to trust it. A complete bilingual user manual now ships inside the app, always available and answerable by the AI itself. The dashboard gains a capability evaluation center, so you can run the Hope Core scenarios against your own models and see how they actually perform on real tasks. And streaming replies are now written to disk before they are broadcast, so a crash, a reload, or a mid-stream stop no longer loses content.

## Highlights

- **A built-in user manual and help center.** The complete bilingual manual now ships with the app and works entirely offline. The sidebar, the About page, the macOS Help menu, and the tray all open a dedicated help window with full-text search, a chapter outline, and in-chapter find. Every settings section has a "?" that jumps straight to the matching chapter, and selecting any passage in the manual lets you ask the AI about it with that context attached. The assistant can read the manual too, so it can answer "how do I do X" questions directly.
- **A capability evaluation center in the dashboard.** Run the Hope Core capability evals against a configured provider, a local model, or your signed-in Codex, covering goal progression, workflows, concurrent async jobs, and sub-agent collaboration. Preview the plan and set cost and duration budgets before starting, watch per-scenario progress live and cancel at any point, then browse per-scenario detail, compare versions, view trends, and attach notes; results import and export as well. Failed scenarios explain the actual spend, the resolved budget ceiling, which dimension tripped, and the causal trace — not just a failure marker. A cancelled or interrupted experiment cannot resume in place, since requests already issued to a provider and their billing cannot be frozen safely; instead, "Retry as new experiment" from History reruns the original request while preserving the parent record and its incurred cost. Starting an evaluation is a desktop operation — Server and Web mode expose redacted read-only queries — and local results are diagnostic only, never release gate evidence.
- **Replies no longer get lost to a crash, a reload, or a mid-stream stop.** Streaming replies are now persisted before they are broadcast: the assistant message, the context, and the usage ledger all settle in a single transaction, and a restart after a crash or force-quit recovers to a consistent state from what was written to disk. Reloading the page no longer drops, duplicates, or rolls back content you already received, and the same recovery path covers total provider failure, an explicit stop, and automatic failover. Incognito sessions still write nothing to disk at any point.
- **A retired embedded inference engine is gone.** The ONNX runtime left over from the old local embedding implementation has been removed and the embedded frontend assets are now brotli-compressed, together trimming roughly 10MB from the installers. Local embedding is unaffected and continues to be served by the Ollama-based option under Settings → Memory → Local embedding helper. To be clear about the net effect: this release's new built-in manual and capability-eval assets are larger than that saving, so installers still come out roughly 7–10MB bigger than v0.20.1.

## Fixes and stability

- **Built-in skills and the browser extension are no longer missing or stale in any deployment shape.** Both are now compiled into the main binary, so the desktop app, Docker, and the single-file build all carry them and pick up updates automatically; the single-file release additionally bundles the `ha-browser-host` extension bridge and swaps it during self-update. After upgrading, an already-loaded Chrome extension needs one manual reload from `chrome://extensions` to take effect.
- **The settings page opens on its first section.** Opening settings from the sidebar gear now lands on Personal settings instead of always jumping to Model settings; entry points that explicitly target a section are unchanged.
- **The sidebar's session loading animation no longer freezes.** Running sessions keep showing their spinner and pending-state pulse after the sidebar first loads, instead of staying frozen until you interact with the sidebar once.
