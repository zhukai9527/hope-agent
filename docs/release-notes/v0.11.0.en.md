# Hope Agent 0.11.0 — Background Jobs as First-Class Citizens · Next-Gen Structured Memory

> [简体中文](https://github.com/shiwenwen/hope-agent/blob/v0.11.0/docs/release-notes/v0.11.0.md) · **English**

> Released: 2026-06-21

> See [`CHANGELOG.md`](https://github.com/shiwenwen/hope-agent/blob/v0.11.0/CHANGELOG.md#0110---2026-06-21) for the full list.

v0.11.0 follows two main threads. The first turns **background work into a first-class citizen** of the product: a unified Background Jobs model, a visible "Background Jobs" panel, concurrency governance with fair queuing, live output while jobs run, batched concurrent subagent delegation, and a wave of robustness fixes around background execution. The second brings **next-generation structured memory (Dreaming / Claims)** to maturity: a structured memory layer carrying evidence, time-sensitivity and confidence is now on by default, paired with Memory Profile synthesis, hybrid retrieval, system-prompt injection, deep consolidation, and a user-correction loop — making long-term memory more reliable, traceable and correctable.

## Highlights

- **Background Jobs panel + header badge + completion notifications**: a new "Background Jobs" panel on the right gives you one place to see this session's running and recent background work — tools, task groups, and background subagents — cancel them one by one or all at once, with task groups showing "N of M done" progress. Local model downloads, embedding, and reindex jobs are mirrored read-only into the same panel, eliminating overlapping task surfaces. The header gains a running-count badge, the Workspace panel offers a condensed at-a-glance block, and a desktop notification fires when a background job finishes while the window is in the background (toggle it off under Settings → Notifications → Background job complete).
- **Background concurrency governance + auto-queuing + per-session fair share**: a new "max concurrent background jobs" setting (defaulting to your machine's logical core count, `0` for unlimited) caps how many background tool jobs run at once, so the model can't exhaust threads and memory by chaining background execution across turns. When the cap is reached, new jobs are no longer rejected — they queue and run in fair, round-robin rotation as slots free up. A separate "per-session concurrency cap" acts as a hard constraint, so no single session or IM group chat, however busy, can monopolize the global pool and starve others. Transient failures can optionally retry with backoff (off by default, applying only to tools that are safe to re-run such as web search), and a new "unattended approval" policy decides whether to block (default) or proceed when no one is available to respond to an approval.
- **Delegate concurrent subagents in batch, collect results in one shot (`batch_spawn`)**: when you dispatch multiple background subagents at once with `batch_spawn`, they run concurrently as a single batch, and once all finish their results are merged into one notification injected back into the conversation (including each subagent's success/failure, task, and result). The model can end its turn right after dispatching — no polling each one — and a single subagent failing doesn't prevent the rest from coming back together. Background subagents are now part of the unified Background Jobs model: you can list / inspect / cancel them via `job_status`, and they auto-queue rather than error when over the concurrency cap. The `job_status` tool itself is upgraded into a multi-job management surface supporting batch `list` / `wait` / `cancel` / `result`. Meanwhile, running background commands can now stream their recent output live via `output_tail`, and an agent can use `schedule_wakeup` to set a one-shot delayed wake-up for itself to wait on external state.
- **Next-gen structured memory (Dreaming / Claims) on by default**: automatic memory extraction now also extracts evidence-backed structured claims (with scope, type, confidence, salience, and time-sensitivity) alongside the legacy memory — in the same LLM call as existing fact extraction, with no extra round-trip. High-importance active claims are injected stably into the system prompt as a "Pinned Memory" section, and legacy shadow memories they cover are deduplicated out of the legacy memory section; active-recall candidates can also be expanded to include structured claims. Structured claims support hybrid retrieval over FTS5 full-text plus vector search (RRF fusion) and are automatically re-embedded when you switch embedding models.
- **Memory Profile synthesis + deep consolidation**: with profile synthesis enabled (on by default), an offline pass synthesizes a readable Memory Profile snapshot from structured claims, per scope — using low-cost rule-based aggregation when idle or on a schedule, and an LLM rewrite when triggered manually — and maintains a "User Profile" section in the system prompt. Memory → Profile offers a read-only view with manual refresh. Dashboard → Memory Dreaming adds "Deep consolidation," which runs time-based expiry, duplicate merging, and conflict detection over structured claims; genuine contradictions are flagged "needs review" for you to confirm, never silently overwritten or deleted.
- **User-correction loop (Lucid Review) + traceable provenance**: Dashboard → Memory Dreaming adds a "Needs review" queue listing structured claims flagged needs_review, which you can approve / edit / reject / mark outdated / move scope / pin / forget one at a time — each action writes an audit decision log and takes effect on the very next prompt. Every memory promoted by offline consolidation now carries provenance hints, shown as source labels in its decision record; every offline run (manual / idle / scheduled) is persisted, so you can review run history and promotion decisions even after restart, and concurrent triggers across processes are serialized via a SQLite lease so the work runs exactly once. There's also a historical backfill entry to register legacy memories as structured claims in bulk (preview with a dry-run first, then execute), plus a new LLM-free deterministic evaluation harness that guards memory safety red lines.

## Other improvements

### Background Jobs

- **Merged completion injection saves billable turns**: when the model bypasses task groups and fires off several `run_in_background` calls back to back, multiple jobs in the same session that complete within a short window (3 seconds by default, `0` to disable) are merged into a single injection back into the conversation — so "encouraging backgrounding" doesn't degrade into "inflating billable turns."
- **Background commands that hit an approval mid-run now park and wait**: an `exec` run explicitly in the background that hits an approval point now parks as an "awaiting approval" background job — once approved it resumes automatically, once denied it ends as "denied," with the outcome injected back via the completion notification and a clock icon shown in the panel. When a background subagent hits an approval point internally, the panel and `job_status` likewise show "awaiting approval."
- **Background job semantics and configurable knobs adjusted**: background jobs and `exec` commands no longer set a timeout by default (the model can still set a per-call limit), and `job_status` is narrowed from active polling to a non-blocking status snapshot. Settings → Async Tools adds four new knobs — running-output retention size, queue capacity, maximum wake-up delay, and per-session wake-up cap (all clamped to safe ranges, with matching read/write support in the `ha-settings` skill).
- **Async tool auto-backgrounding off by default**: `autoBackgroundSecs` now defaults to `0` instead of 30 seconds, so new configs no longer auto-background a synchronous tool just because it runs long; explicit `run_in_background` and the per-agent "always run in background" policy are unaffected. To keep the old behavior, set a positive number of seconds under Settings → Async Tools → Auto-background threshold.

### Background Jobs robustness

- **More reliable background result injection**: in self-hosted / IM / ACP modes, background results no longer collide with an in-progress turn (the "queue while busy, inject when idle" logic now lives in the shared chat engine used by all entry points); when a parent session is busy for a long stretch, results are no longer mis-marked as "injected" and permanently dropped, but held over for the next startup to deliver; and custom hooks' `PostToolUse` / `PostToolUseFailure` now fire when a background async tool actually completes / fails / times out / is cancelled.
- **Background job cancellation and orphan cleanup**: cross-process cancellation of a background job now truly aborts the other side's job rather than just changing status; cancelling a conversation also reclaims background jobs that happened to be approved and launched within the finalize window; and orphaned background `exec` child processes left behind after a crash or force-quit are detected and group-terminated on the next startup.
- **Many read-only tools in one turn no longer all run at once**: when a single assistant reply contains many read-only tool calls (e.g. multiple `web_fetch`), at most 8 now run concurrently and the rest queue automatically — avoiding a burst of outbound requests or exhausting file descriptors all at once.
- **Subagent concurrency cap raised by default + per-agent configurable**: the default cap on subagents running concurrently per parent session is raised from 5 to 8, and it can be tuned per agent under Settings → Agent → Subagents → Max concurrency (range 1–50; previously this setting had no effect).

### Approvals and permissions

- **Smoother smart approvals**: in Smart mode, re-editing a file you've already changed in this conversation no longer re-prompts; the first write / edit to a file within the working directory is judged by the model on risk (routine changes pass, high-risk actions like sweeping overwrites or deletions still confirm), while files outside the working directory are treated more conservatively; protected paths and dangerous commands still always require confirmation, no matter what.
- **Approval-timeout "proceed" no longer punches through dangerous operations**: when the approval-timeout action is set to "proceed," high-risk reasons such as dangerous commands or writes to protected paths are always denied on timeout ("proceed" applies only to ordinary approvals); approval timeouts now also dismiss the dialog on all endpoints just like a normal response.
- **More consistent, safer multi-endpoint approvals**: once an approval is handled on any endpoint (desktop / web / IM), the dialogs on the others dismiss automatically; IM approval responses strictly validate their source, so a cross-session button tap or text reply can't approve an approval that isn't its own, and when a session is taken over by another endpoint, its outstanding approvals are automatically denied.
- **Browser script execution wired into the unified permission engine**: `browser.control.evaluate` no longer bypasses the session permission mode with its own `ask_user_question`, but enters the unified tool-approval chain as a soft approval (Default prompts, Smart can auto-pass, Yolo passes directly), while still keeping the SSRF literal scan.
- **Permission-mode switcher can update the agent default**: the permission-mode menu above the chat input gains an "also update agent default" toggle, so switching `default` / `smart` / `yolo` also updates the current agent's default permission mode for new sessions.
- **Unattended entry points no longer hang forever on an approval**: scheduled jobs, headless services with no connected client and no IM session, ACP clients without a permission channel, and headless subagents that trigger a tool requiring approval are now handled immediately per the "unattended approval" policy (fail-closed deny by default, with a clear root cause) instead of hanging; the CLI setup wizard's "no approval needed" option has also been corrected to be equivalent to enabling YOLO.

### Context compaction

- **Manual compaction behaves consistently across desktop / HTTP server / IM**: `compact_context_now` and `/compact` no longer return "unsupported" or do only a synchronous cleanup in HTTP / server mode — they now restore history per the session's model, bypass throttling, and run Tier 3 summarization where possible; on desktop, compaction no longer temporarily clears the global agent, and the related prompts now have full multilingual copy.
- **Manual compaction actually trims oversized recovered tool results**: the compaction buttons in the `/compact` and `/context` cards now show this run's result, and manual compaction clears the oversized tool results from interrupt-recovered turns — so image-viewing and tool outputs that were duplicated on recovery don't leave compaction finishing with barely any token reduction.
- **Live context usage refresh during long tool turns**: a long turn with tool calls now updates the context-usage bar after every API round-trip, instead of jumping only when the whole turn finishes.
- **Very long turns can compact context between tool rounds**: a very long turn with many tool calls now sets compaction checkpoints between tool rounds, compacting in place before history overflows the context window instead of waiting for the next user message; the summary also more reliably preserves failed attempts, user corrections, background tasks, and recently-edited-file hints.

### Visuals and images

- **The image-viewing tool now previews the image directly**: the `image` tool card now shows, below it, the actual image handed to the model for this call, making it easy to verify which image the model is looking at; incognito or no-disk scenarios still don't generate a preview attachment. The tool's copy has also been unified from "image analysis / visual input" to "view image," with UI labels, settings copy, and all 12 languages updated.
- **Sending an image to a non-vision model no longer fails the whole turn**: when you send an image to a text-only model, it now silently drops the image, answers as usual, and attaches a small gray note in the conversation explaining what happened (IM channels get a matching notice) — instead of failing the entire turn because the backend rejected the image content.
- **Vision tool results no longer leak internal markers when an image is unreadable**: when a materialized image file is unreadable / too large / of an unsupported type and can't be encoded as provider vision input, it degrades to a text description instead of feeding internal markers back to the model (aligned across the Anthropic / OpenAI Chat / Responses paths).

### Sidebar and project experience

- **Sidebar session list defaults to simple mode**: the session list now defaults to a single-line compact view — tighter, with more sessions visible per screen (switch back to detailed mode in settings); in simple mode, hovering a session item fades in its agent's avatar and name in place, and each session's context menu gains a "Delete" item.
- **Independent pagination for project sessions in the sidebar**: each project's session list now loads independently, showing the 15 most recent by default; "Show more" at the bottom loads 15 more each time, "Collapse" returns to 15 — no longer bounded by the global pagination window, so older sessions inside a project become visible.
- **Sidebar project area and collapse animation polish**: project groups now feel more like first-class objects in the sidebar — project names are bold with reserved space for action buttons, and the count badge no longer crowds the title; expanding and collapsing of agents / projects / archived projects / in-project session groups all use a smooth animation.
- **Default permission mode set inside a project now takes effect correctly**: previously, entering a project immediately created an empty session, bypassing the path that initializes from the agent's default permission mode — so after setting a permission mode as default inside a project, newly opened project conversations still showed `default`. Now "new conversation" inside a project behaves like an ordinary conversation, creating the session only when the first message is sent, with the backend uniformly initializing from the agent's default permission mode.

### Robustness

- **A config read failure no longer silently wipes the entire config**: previously, when `config.json` failed to read or parse, it would silently fall back to the default config, and the very next save would overwrite the defaults back to disk — permanently wiping your configured providers / MCP servers / initial-setup state and forcing the setup wizard on every launch. Now a read failure is clearly logged as an error and the unparseable file is backed up to `config.json.corrupt-<timestamp>`; if the config is still unreadable at startup, a recovery screen lets you retry or restore an automatic backup, and a UTF-8 BOM written by Windows editors is tolerated.
- **Fixed desktop auto-update failing when the app lives on a non-system volume**: when Hope Agent is installed / running on a disk volume different from the system temp directory, desktop "Update and restart" previously failed mid-way with a cross-device move error (`Cross-device link`); the update's temp directory is now redirected at startup to the app's own volume, so updates complete normally.
- **Reasoning-depth setting is easier to discover**: when using a reasoning-capable custom model, settings now add an explanation next to the model's "supports reasoning / thinking" toggle; and when a model isn't reasoning-enabled, the chat model menu points you to Settings → Model Providers to enable it — so you no longer mistakenly think custom providers can't adjust reasoning depth.
- **Other fixes**: the interrupt-recovery prompt no longer flashes repeatedly; completed steps within a long turn collapse in real time, and the input box restores automatically after an interruption; incognito sessions' "burn on close" is now more thorough (large tool results and background results no longer hit disk, plaintext commands are no longer stored, and "always allow" is no longer offered); rapid-fire IM messages no longer step on each other (one turn is processed at a time, with later messages held for the next turn); the profile settings entry no longer shows an i18n key error; and the memory-claim backfill dialog no longer overflows horizontally.

### Other UI refinements

- **Responsive and animation polish for the chat side panels in narrow windows**: as the window shrinks, the right Workspace panel collapses first and the left session sidebar collapses next; when the window widens again, only panels collapsed by the responsive path are restored, without overriding the user's manual choice; the auto-collapse threshold now uses a `matchMedia` listener to reduce re-render pressure while dragging the window, and the side-panel animations are split into a layout-placeholder transition and a `transform` / `opacity` visual slide.
- **The desktop main window remembers and restores its last position**: the desktop app now remembers the main window's position alongside its size and restores where you last left it on restart; if the saved position lands on a disconnected display, it is clamped back onto an available screen.
- **The memory Claims settings page is easier to understand**: Memory → Claims adds a user-facing explanation section covering what claims are, which states affect AI replies, how to review them, and what "backfill historical memory" does; it also unifies Simplified / Traditional Chinese terminology and renames the backfill entry to the clearer "backfill historical memory."
