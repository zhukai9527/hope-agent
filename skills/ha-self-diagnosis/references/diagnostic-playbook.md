# Diagnostic Playbook

## Bug or Failure

1. Capture the user's symptom in one sentence.
2. Determine run mode: desktop, server, ACP, IM channel, cron, subagent, or MCP.
3. Locate the subsystem in the **Subsystem Reference** below — it names the doc, entry module, the right database, and the stable log category to grep.
4. Check settings relevant to the subsystem with `get_settings`.
5. Query recent logs read-only (the log `category` for each subsystem is in the reference):

```bash
sqlite3 -readonly -cmd ".headers on" -cmd ".mode column" ~/.hope-agent/logs.db \
  "SELECT timestamp, level, category, source, message, details
     FROM logs
    WHERE level IN ('ERROR','WARN')
    ORDER BY timestamp DESC
    LIMIT 80;"
```

   Narrow by subsystem with `AND category = '<category>'` (e.g. `knowledge`, `hooks`, `browser`, `failover`, `cron`).

6. If the issue is session-specific, inspect messages/tool rows:

```bash
sqlite3 -readonly -cmd ".headers on" -cmd ".mode column" ~/.hope-agent/sessions.db \
  "SELECT timestamp, role, tool_name, is_error, substr(content,1,240) AS content,
          substr(tool_result,1,240) AS tool_result
     FROM messages
    WHERE session_id = '<SESSION_ID>'
    ORDER BY timestamp DESC
    LIMIT 80;"
```

7. Map symptoms to source files using the **Subsystem Reference** and `docs/architecture/`.
8. Check whether a newer release may already fix it with `app_update(action="check")` when appropriate.
9. For issue reporting, include only relevant, redacted evidence.

## Subsystem Reference

Per subsystem: the architecture doc, the diagnostic entry module(s), the
runtime state (DB / config), the stable `logs.db` `category` to grep, and the
gotcha that most often explains a failure. Open every DB read-only.

### Subagent / Team / Cron — `subagent.md`, `agent-team.md`, `cron.md`

- Entry: `subagent/injection.rs`, `team/coordinator.rs`, `cron/scheduler.rs`, `cron/executor.rs`, `session/subagent_db.rs`, `wakeup/mod.rs` (+`wakeup/db.rs`).
- State: `cron.db` (`cron_jobs` + `cron_run_logs` + `cron_meta`), `sessions.db` (`subagent_runs`, `teams`/`team_*`), `background_jobs.db`, `wakeups.db` (agent self-scheduled `schedule_wakeup`). Config: `subagents.enabled`/`max_spawn_depth`, `team.*`, `cron.{maxConcurrent,jobTimeoutSecs,atGraceSecs}`. Cron-panel conversation surface: `cron_run_timeline` / `cron_unread_total` / `cron_mark_all_read` (Tauri + HTTP `/api/cron/{timeline,unread,read-all}`) + event `cron:unread_changed`; the cross-job timeline is assembled in `cron::timeline` (joins `cron.db` run logs with `sessions.db` title/unread in Rust — the two DBs can't be SQL-joined).
- Grep: `category IN ('subagent','team','cron','wakeup')`.
- Gotcha: cron `status='disabled'` ← `consecutive_failures>=max_failures` (default 5, **genuine run failures only** — infra errors like session-create-fail route through `reschedule_without_failure` with `count_toward_disable=false` and do NOT bump the counter, so transient DB blips can't disable a healthy job); non-null `running_at` after restart = orphan (`clear_all_running` clears it; in-process panics covered by RAII `RunningMarkerGuard`). `cron_run_logs.status` is now **7-valued** (`classify_cron_terminal` terminals + the no-session infra path): `running` (in-flight / crash breadcrumb — `recover_orphaned_runs` finalizes stale ones to `error` on startup), `success`, `empty` (non-cancelled zero-output `Ok` — delivery skipped / no empty message sent, schedule still advances; **recurring `empty` does NOT pop a desktop toast** — `notify_empty = notify_on_complete && At`, only one-shot `At` notifies), `cancelled` (cancel interrupt — no empty delivery, schedule NOT advanced; one-shot `At` terminalized to `completed`), `timeout` (exceeded the per-run budget `CronConfig.job_timeout_secs`, default 600, clamp `[30,7200]`, per-job `CronJob.job_timeout_secs` override — **not a fixed 300s anymore**), `error`, `no_session` (infra failure: session creation failed / turn never ran — counts as failure in the dashboard denylist but routed via `reschedule_without_failure`, so it does NOT bump `consecutive_failures` / auto-disable). Concurrency: `CronConfig.max_concurrent` (default 5, `0`=unlimited) caps in-flight runs; the scheduler grabs a slot **before** claiming (slot-before-claim), so jobs beyond the cap keep `next_run_at` and retry on the next 15s tick (not skipped/lost) — a job not firing under a full pool is starvation, check `count_running()` for leaked `running_at` markers. A scheduled `success`/`empty` run releases its slot (`clear_running`) **before** the IM delivery (so a hung delivery target can't pin a cap slot); run-now keeps its slot through delivery. **run-now (`execute_job_public`, all 3 entry points: Tauri / HTTP / `manage_cron`) is Primary-only** — a non-Primary instance refuses it (returns an error, not a fake success) and `mark_missed_at_jobs` now runs every tick before dispatch (not just at startup). `cron_meta` KV holds `scheduler_heartbeat` (stale ≥300s on startup → `app_warn` "scheduler was offline ~Ns"; Primary-only, crash recovered via catch-up on restart) + one-time `tz_backfill_done` sentinel (gates host-timezone backfill of null-tz `Cron` rows; absent = backfill will re-run, host-undetectable = sentinel not written so it retries). `subagent_runs.child_agent_id` prefix `tool_job:` = async-tool injection (not a real subagent), `team:` = team member; stuck `Running` cleaned by `cleanup_orphan_runs`. **R6 subagent projection**: a user-delegated background subagent run ALSO gets a one-way `background_jobs` row (`kind='subagent'`, `subagent_run_id` FK) so it shows in `job_status list/cancel` — `subagent_runs` is the truth source (result/error live there; the projection has none). Projection status drifting from `subagent_runs.status`? sync is hooked in `update_subagent_status` (the choke point) — the raw-SQL `cleanup_orphan_subagent_runs` bypasses it, so after a restart a projection may read `interrupted` (via `replay_pending_jobs`) while the run reads `error` (cosmetic; projection is just a view). Cancelling a `kind=subagent` job routes to `subagent::request_cancel_run`, NOT the tool-job cancel path. Internal plan/team/hook subagents (`skip_parent_injection=true`) and incognito-parent runs are NOT projected. **R7.2 reject→queue**: a subagent that hits the per-session concurrency limit (`max_concurrent_for_agent`, default 8) now PARKS as `subagent_runs.status='queued'` (projection `kind=subagent` status `queued`) instead of erroring — held in the in-memory `subagent::queue` (live `SpawnParams`, lost on restart). The per-process `run_subagent_scheduler` promotes it (`queued→spawning`) when a running child settles (woken from the `update_subagent_status` terminal choke point) + a 5s fallback tick. Stuck `queued`? check the session is actually under its limit (`count_active_subagent_runs` counts only `spawning`/`running` — `queued` is excluded by design so it can't deadlock its own promotion) and that a sibling actually reached terminal to fire the wake; restart sweeps `queued` rows to Orphaned (`error`) via `cleanup_orphan_subagent_runs`. Structural limits (depth/batch/agent-exists) still hard-reject, never queue. A `kind=subagent` projection sitting at `awaiting_approval`? that label is set by the **R8-follow-up `async_jobs::approval_projection_watcher`** — it subscribes to `approval_required` / `approval:resolved`, maps the approval's child session → run via `SessionDB::find_active_run_by_child_session` → projection, and flips `running⇄awaiting_approval` (pure label, never gates execution; the inner approval still block-and-waits). Stuck at `awaiting_approval` after the approval was answered? resume only fires on the `approval:resolved` event — check the EventBus delivered it (watcher lag logs under `category='async_jobs'`, source `approval_projection`); `subagent_runs.status` is the truth (the label is a view). **R5 Group fan-out**: `batch_spawn` creates a `kind=group` coordinator row + N `kind=subagent` children sharing `group_id`; the children **suppress their individual injection** and the group fires ONE merged `<task-notification>` when all children settle (single-winner CAS `claim_group_completion`). Batch ran but no merged notification arrived? check the group row: `args_json={"sealed":true}` must be set (set after the spawn loop — if a malformed task `?`-returned before R5's pre-validation fix the group would never seal) AND all children terminal; the merged message reuses the `subagent_result` meta (agent_id=`batch`, run_id=`<group_id>`). Group rows are `injected=1` (own merged injection, never the tool-replay path); restart leaves an in-flight group `interrupted` with no merged notification (child results still in `subagent_runs`). Cancelling a `kind=group` marks the group terminal FIRST then cancels children (so the join CAS loses — no injection for a cancelled batch). `schedule_wakeup` fires go through `inject_and_run_parent` directly and **do NOT create a `subagent_runs` row** — they appear as a `ParentInjection`-source user message tagged `attachments_meta={"wakeup_trigger":{run_id}}`. Wakeup never fires → `wakeups.db` row exists (delivered rows are deleted) but no live timer (arming process exited; re-armed Primary-only on next restart) OR parent never went idle (injection `Abandoned`, left for replay); incognito wakeups are in-memory only (no row). **Cron run sessions are hidden from the main sidebar / global search** (`list_sessions_paged_for_sidebar` passes `exclude_cron=true` → `is_cron=0`; the generic `list_sessions_paged` used by awareness/tray is unaffected) — they live in the cron panel's **History** view (newest-first). "My cron ran but its conversation isn't in the sidebar" is BY DESIGN, not a bug. The cron sidebar-icon unread badge = aggregate unread `assistant` messages across all `is_cron=1` sessions (`cron_unread_total`); "mark all read" calls `mark_all_cron_sessions_read` and emits `cron:unread_changed`. Run-history list paginates via `get_run_logs(limit, offset)` with `limit` clamped ≤200 at both shells.

Panel step-timeline: `tool_actions.rs` (per-session in-memory ring, `browser:action` / `mac_control:action` events, `tool_recent_actions` owner query). macOS TCC layer: `permissions.rs` (`PERMISSION_DEFS` catalog, `check_item` / `request_item` / `folder_status`).
- State: no SQLite — `browser-profiles/` + `browser/{managed-runner,user-attach,runtime}/`, `mac-control/snapshots/` + `mac-control/diagnostics/`. Config: `AppConfig.browser` (`launchCircuit`, `profiles[*]`). macOS permissions hold no persistent state — they probe live TCC each call.
- Grep: `category IN ('browser','mac_control','permissions','tool_actions')`.
- Gotcha: desktop-only — `mac_control` returns `supported=false` off macOS-Tauri and needs live TCC perms (Accessibility + Screen Recording); start with `action='status'`/`'permissions'`. Browser launch failures → `launch_circuit.rs` trip + 3-tier fallback. Missing `browser:frame`/`mac_control:frame` EventBus frames = capture failed. Stale `el_N`/`ref_id` only valid within their originating snapshot. Panel timeline (`tool_actions`) is memory-only — empty after restart is normal, not data loss; entries are purged on session delete/burn, typed text is redacted to `text(N chars)` by design. **macOS TCC (`permissions.rs`)**: `request_item` dispatches by **`def.id`** (NOT `request_mode` — that field is catalog metadata for the frontend only); real branches are the NativePrompt ids, the two `automation` ids (osascript probe), and a `_` fallback (`open_settings_pane` + re-check, covering OpenSettings + None). Folder ids carry a `_folder` suffix (`desktop_folder`/`documents_folder`/`downloads_folder` — bare `desktop` etc. do not exist). NativePrompt items already past `NotDetermined` open Settings instead of re-prompting (`requestable_status_or_open`). Off macOS the whole module is a `cfg` stub returning unsupported.

### Ask-user / Prompt system / Media generation — `ask-user.md`, `prompt-system.md`, `media-generation.md`

- Entry: `tools/ask_user_question.rs`, `ask_user/questions.rs`, `channel/worker/ask_user.rs`, `system_prompt/build.rs`, `media_gen/` (unified image/audio generation: provider→models→chains, executor, catalog), `tools/{image_generate,audio_generate}/` (chat-tool front-ends).

Config: `ask_user_question_timeout_enabled` (default false = wait forever) / `_secs`, `mediaGen` (`providers[]` + per-function `chains` — image / speech / music / sfx, each = primary model + fallbacks; an empty chain falls back to provider order — plus `imageDefaults` / `audioDefaults`).

Image / audio generation failures return a transparent per-candidate `failover_log` (one line per provider+model tried in the function chain, including capability-mismatch and endpoint-blocked skips). ### Reliability / Security / Sandbox / Platform / Backup — `reliability.md`, `security.md`, `sandbox.md`, `platform.md`, `backup-autosave.md`

- Entry: `guardian.rs`, `self_diagnosis.rs`, `crash_journal.rs`, `security/ssrf.rs`, `security/dangerous.rs`, `platform/mod.rs`, `docker/`, `backup.rs` (`create_backup` / `snapshot_before_write` / `restore_*`).
- State: `crash_journal.json` (not SQLite), `backups/backup_{ts}/` (full crash backups, keep `MAX_BACKUPS`=5) + `backups/autosave/` (per-write single-file snapshots, keep `MAX_AUTOSAVES`=50). Config: `guardian.enabled`, `permission.global_yolo` (+ CLI `--dangerously-skip-all-approvals`), `ssrf{}`, `searxng_docker_use_proxy`.
- Grep: SearXNG / Docker sandbox → `category='sandbox'` (source `docker` / `searxng`); permission decisions → `category='permission'`. Crash recovery / Guardian / `self_diagnosis` have **no** `logs.db` category — read `crash_journal.json` directly (the primary signal). Backup failures are warn-only under `category='config'`-adjacent paths (never block a legit write).
- Gotcha: crash/restart loop → read `crash_journal.json` (`crashes[].signal`, last `diagnosis_result`); Guardian does not supervise `server`/`acp`. Dangerous-skip must be checked via `security::dangerous::is_dangerous_skip_active()` (CLI flag OR `permission.global_yolo`), never the raw field. All outbound HTTP must pass `ssrf::check_url`. **Backup (`backup.rs`)**: two budget-isolated mechanisms — `auto_fix` restores a corrupt `config.json` from the **full crash backup** (`list_backups().first()`), NEVER from autosave; autosave is per-write single-file undo (filename encodes `{ts}__{kind}__{category}__{source}`, parsed by `splitn(4,"__")` — the `__` separator is a stable contract). `snapshot_before_write` swallows all errors (warn-only) so a broken safety-net can't block the user's write. The `updater/backup.rs` (bare-binary store/prune) is a **completely separate** module — don't conflate.
- **Backend freeze / "everything loading forever" (issue #433 Bug 2)** → `category='blocking'` source `run_blocking`. Every SQLite DB + config write is synchronous under a `Mutex`/write-lock; async Tauri commands / ha-server handlers MUST offload them via `ha_core::blocking::run_blocking` / `SessionDB::run` / `config::mutate_config_async` (`crates/ha-core/src/blocking.rs`, Layer C′ in `process-model.md`) — an inline sync DB/config call in an `async fn` pins a tokio worker for the whole lock-wait + file IO, and on a stalled filesystem (AV scan / cloud-synced home / slow disk) starves all `num_cpus` workers so the process stays alive but every command hangs. Any blocking op >5s logs a `blocking` warn naming the caller's closure (the culprit's module/function). Symptom: send-message spins + settings page stuck loading, no crash. If it reproduces, grep `category='blocking'` for the slow closure; also verify `~/.hope-agent` isn't on OneDrive/iCloud and no new `async fn` reintroduced an inline `.append_message(`/`mutate_config(`/DB call (the AGENTS.md red line).
